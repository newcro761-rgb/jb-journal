# 매매내역 수정 + 예수금 기반 수익률 — Design

## Problem

매매내역 탭은 지금 삭제(✕)만 가능하고 수정이 안 된다. 결과 입력 중 가격/수량/날짜를 잘못 적어도 고칠 방법이 없다(삭제 후 재입력만 가능한데, 그러면 원래 생성 시점/순서가 흐트러진다). 또한 "평균 수익률" 통계는 거래별 수익률을 단순 평균한 값이라, 실제 투입한 자본(예수금) 대비 전체 수익률을 보여주지 못한다.

## Goals

- 매매내역의 각 거래를 종목명/종목코드와 매도 내역(가격/수량/날짜/세금/메모)까지 수정할 수 있게 한다. 매도 건은 추가/삭제도 가능하다.
- 수정 시 수익(`profit`), 수익률(`rate`), 상태(`status`)를 자동으로 재계산한다.
- 예수금(투입 자본)을 입력할 수 있게 하고, 설정되면 "평균 수익률" 대신 "자본 기준 수익률"(총 수익금 / 예수금)을 보여준다.

## Non-Goals

- 매수 정보(`avg`, `totalQty`, `use2`, 매수일)와 장세(`regime`)는 이번 수정 대상에서 제외한다 — 사용자가 요청한 건 "잘못 적은 매도 정보"의 정정이고, 매수 구조까지 바꾸는 건 범위 밖이다.
- 이미 Google Sheets에 올라간 row는 수정하지 않는다 — Sheets 자체 수정은 기존 "시트 열기" 탭에서 사용자가 직접 한다.
- 예수금에 입출금 이력을 따로 추적하지 않는다 — 단일 숫자값을 사용자가 필요할 때마다 직접 고쳐 쓰는 방식이다.
- 1차+2차 혼합 매도 건을 수정할 때 원래 계산에 쓰인 1차/2차 개별 기준가를 복원하지 않는다 — 이미 저장된 합산 평단가(`avg`) 하나로 통일해서 재계산한다(아래 Data Model 참고, 사용자 확정).

## Data Model

### 예수금

새 `localStorage` 키 `jb-capital` — 숫자 하나(문자열로 저장). 설정 안 됨(빈 값/0)이면 기존 평균 수익률 로직으로 폴백한다.

### 매매내역 수정 시 재계산

`hist` 항목의 `sells` 배열을 수정 모달에서 고친 뒤, 아래 공식으로 `profit`/`rate`/`status`를 다시 만든다. 기준가는 항상 `avg`(이미 저장되어 있는 합산 평단가) 하나만 쓴다 — 1차만 매도된 거래는 원래도 `avg === b1`이라 정확하고, 1차+2차 혼합 거래는 원래 계산(`b1`/블렌드 평균 분리 계산)과 살짝 다른 근사값이 될 수 있다(사용자 확정, Non-Goals).

```js
const totalTax = sells.reduce((s, x) => s + (x.tax || 0), 0);
const profit = Math.round(sells.reduce((s, x) => s + (x.price - avg) * x.qty, 0) - totalTax);
const totalQty = sells.reduce((s, x) => s + x.qty, 0);
const rate = parseFloat((profit / (avg * totalQty)).toFixed(4));
const bestTag = sells.reduce((best, s) =>
  TIER_RANK.indexOf(s.tag) > TIER_RANK.indexOf(best) ? s.tag : best,
  sells[0].tag);
const status = (profit >= 0 ? '수익' : '손절') + `(${sells.length}차분할) · 최고 ${bestTag}`;
```

매도 row마다 `tag`도 같은 기준(`avg`)으로 다시 계산: `tierTag((price - avg) / avg)`.

`totalQty`도 재계산된 값으로 같이 갱신한다(매도 row가 추가/삭제되면 원래 `totalQty`와 달라질 수 있음).

## UI

### 예수금 입력

매매내역 탭, `stat-grid` 바로 위에 입력 필드 추가:

```html
<div class="section" style="padding:10px 14px;">
  <label>예수금(원)</label>
  <input type="number" id="capitalInput" placeholder="10000000" oninput="saveCapital()">
</div>
```

페이지 로드 시 `localStorage`에서 읽어 채워 넣는다. 입력할 때마다 바로 저장(별도 저장 버튼 없음, 기존 계산 탭의 실시간 입력 패턴과 동일).

### 통계 카드 — "평균 수익률" → "자본 기준 수익률"

예수금이 설정돼 있으면(>0) 카드 라벨이 "자본 기준 수익률"로 바뀌고 값은 `총 수익금 / 예수금 × 100`. 설정 안 돼 있으면 기존처럼 "평균 수익률"(거래별 rate 단순 평균) 그대로.

### 수정 모달

각 `hist-item`에 기존 "✕"(삭제) 옆에 "✎"(수정) 버튼을 추가. 클릭하면 새 모달(`edit-modal`)이 열리고:

- 종목명, 종목코드: 텍스트 입력
- 매도 row들: 결과입력 모달과 같은 필드(매도가/수량/매도일/세금/메모) + 추가("+ 매도 추가")/삭제 버튼
- "저장하기" 클릭 시 위 재계산 공식을 적용해 `hist` 배열의 해당 항목을 갱신하고 `localStorage`에 저장, 매매내역 화면을 다시 그린다
- 매도 row가 0개인 채로 저장하려 하면 토스트로 막는다(기존 결과입력 모달의 "매도가/수량/매도일을 입력하세요." 검증과 동일한 철학)

## Backward Compatibility

- 예수금이 한 번도 설정되지 않은 사용자는 화면이 지금과 동일하게 보인다("평균 수익률" 그대로).
- 기존 `hist` 항목 중 `sells`의 일부 항목에 `tag` 필드가 없는 경우(이 기능 이전에 생성된 거래)도, 수정 모달을 열어 저장하면 그 시점에 모든 row의 `tag`가 새로 계산되어 채워진다 — 수정하지 않고 그냥 두면 기존처럼 태그 없이 표시된다(매매내역 탭 자체 렌더링은 변경 없음).
- `status` 문자열에 `· 최고 {tier}`가 없던 옛 거래(이 기능 이전, 또는 Group2 이전에 만들어진 거래)를 수정하면, 저장 시 `status`가 새 포맷(`· 최고 {tier}` 포함)으로 통째로 다시 생성된다 — 수정은 항상 최신 포맷으로 재작성됨을 의미하며, 이는 의도된 동작이다.
